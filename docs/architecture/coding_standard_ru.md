# Coding Standard V2

## Общие правила

- Пишем простой и читаемый код.
- Не создаем монолитные модули.
- Не используем универсальные свалки вроде `utils.py` и `helpers.py`.
- Один файл отвечает за одну зону логики.
- Один модуль не должен делать несколько независимых вещей.

## Имена

- Имена файлов и модулей: `snake_case`, ASCII.
- Имена функций и переменных: английские.
- Комментарии и смысловые пояснения: на русском.

## Комментарии

- Комментарии оформляем через `#`.
- Комментарии пишем только там, где есть:
  - физическое допущение;
  - формула;
  - порог;
  - неочевидная логика;
  - важное инженерное ограничение.
- `""" ... """` как комментарии не используем.

## Качество

- Код должен проходить `ruff`.
- Код должен проходить `mypy`.
- Изменения должны сопровождаться нужными тестами.
- Новый слой не должен тянуть лишние зависимости без причины.

## Диагностика и источники

- Не лечим симптомы, если не определена корневая причина.
- Перед исправлением сначала воспроизводим проблему.
- Если проблема относится к библиотеке, инструменту, конфигурации, протоколу или API, сначала проверяем официальный источник.
- Не выдумываем поведение фреймворка или инструмента, если его можно проверить по документации.
- Исправление должно объясняться через причину, а не через случайный набор правок.

## Микро-задачи

- Каждая микро-задача начинается с краткой формулировки проблемы.
- Для каждой нетривиальной правки фиксируем, что было причиной сбоя или предупреждения.
- Если решение зависит от внешнего инструмента, в закрытии микро-задачи должна быть ссылка на проверенный официальный источник или явная пометка, что причина подтверждена локально.
- Пишем и проверяем код связкой:
  - написали блок;
  - сразу локально проверили этот блок;
  - при необходимости сверились с официальной документацией;
  - только потом продолжаем.
- После завершения модуля повторно проверяем уже весь связанный контур, а не только последнюю строку правки.

## Архитектурная дисциплина

- Side effects допускаются только в слоях `db`, `cli`, `io`, `artifacts`.
- Доменная логика должна быть чистой и тестируемой.
- Ноутбуки не являются источником истины для бизнес-логики.
- Точка входа `main.py` не должна постепенно превращаться в новый монолит.
- Если CLI растет, parser registration и command-specific handlers можно выносить в отдельные модули по ответственности.
- При этом не создаем лишние фасады и абстракции, если достаточно простого разнесения функций по файлам.
