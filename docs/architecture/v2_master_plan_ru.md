# Master Plan V2

## Цель

`V2` строится как новая модульная версия проекта для:

- классификации звезд;
- сравнения нескольких моделей;
- расчета host-like сигнала;
- ранжирования целей для последующих наблюдений.

Следующая крупная волна взросления проекта описана отдельно:

- [v2_maturation_plan_ru.md](/Users/evgeniikuznetsov/Desktop/dspro-vkr/docs/architecture/v2_maturation_plan_ru.md)

## Главные решения

- `V1` не развивается в git и хранится только локально.
- БД остается внешним `read-only` контрактом.
- Архитектура строится не вокруг одной модели, а вокруг общего benchmark-контура.
- Комментарии в коде пишем на русском через `#`.
- Код должен быть простым, модульным и читаемым.

## Базовый pipeline

`quality/ood -> luminosity_or_stage -> spectral_class -> spectral_subclass -> host prior -> observability -> ranking`

## Базовый набор моделей

- `Gaussian V1` как baseline;
- `GMM` как вероятностный кандидат под подклассы;
- `HistGradientBoosting` или `RandomForest` как табличный baseline;
- `MLP` как нейросетевой baseline.

## Правило Для Subclass-Волны

- Перед началом отдельной работы над `spectral_subclass` не начинаем обучение сразу.
- Сначала делаем `data audit` по локальным таблицам БД:
  - хватает ли объектов по каждому нужному подклассу;
  - достаточно ли плотное покрытие внутри целевых спектральных классов;
  - нет ли очевидного шума или перекоса в разметке.
- Если локальных таблиц хватает, используем их и не дергаем `Gaia` без необходимости.
- Если локальных таблиц не хватает, отдельно идем в `Gaia` и выгружаем нужные выборки по конкретным подклассам.
- Только после этого подключаем или пересобираем `spectral_subclass` training source.
- Архитектура `V2` считается уже подготовленной под этот шаг, поэтому это расширение делаем как новую data/task-ветку, а не как новый rewrite.

## Правило Для MK-Разметки

- Не храним полный MK-label как одну строку вида `G2V` внутри ядра модели.
- Храним и обрабатываем звезду по составным частям:
  - `spectral_class` как `O/B/A/F/G/K/M`;
  - `spectral_subclass` как `0..9`;
  - `luminosity_class` как отдельное поле;
  - `peculiarity_suffix` как опциональное поле и не тащим его в первую волну.
- На первой волне целевой label-contract строим вокруг:
  - `spectral_class`;
  - `spectral_subclass`;
  - `luminosity_class` или его укрупненной группы;
  - `quality_state` и `ood_state`.
- Строковые формы вроде `G2V` допустимы только как внешний import/export формат, а не как основной внутренний target.

## Правило Для Quality Gate И OOD

- `RUWE` считаем ранним quality-сигналом, а не вторичным декоративным признаком.
- До основной классификации проверяем:
  - `ruwe`;
  - `parallax_over_error`;
  - наличие критически важных признаков;
  - базовую физическую правдоподобность значений.
- Объекты с плохим качеством не должны молча попадать в обычный scoring:
  - либо переводим их в `unknown`;
  - либо отмечаем как `OOD`;
  - либо жестко понижаем приоритет.
- Правило порога для `RUWE` берем не из догадки, а из литературы и практики Gaia, с явной фиксацией в docs и коде.

## Правило Для Внешнего Spectral Source

- Если локальные таблицы БД не дают достаточного покрытия по `spectral_subclass` и `luminosity_class`, идем не только в `Gaia`, но и во внешние спектральные каталоги.
- Для массовой MK-разметки рассматриваем связку:
  - внешний spectral catalog;
  - crossmatch с `Gaia`;
  - загрузка результата в локальную БД;
  - только после этого обучение.
- Этот шаг считаем отдельной data engineering-волной, а не ad hoc загрузкой "по ходу".

## Правило Для Металличности

- `metallicity` не делаем магическим универсальным скорингом "чем больше, тем лучше" или наоборот.
- Металличность идет отдельным научным блоком:
  - собираем первичные источники;
  - фиксируем, для каких типов планет связь подтверждена;
  - выбираем форму priors только после этого.
- На текущем этапе считаем:
  - для газовых гигантов связь с высокой металличностью, вероятнее всего, положительная;
  - для малых и каменистых планет зависимость слабее и не должна задаваться грубой ручной эвристикой.
- До отдельного review по литературе металличность используем осторожно как дополнительный фактор, а не как доминирующий фильтр.

## Завершенный Cleanup Перед Следующей Волной

- Cleanup выполнен и закрыт:

## Тестовый Аудит На Текущем Этапе

- Все Python-тесты, которые должны собираться, реально собираются и исполняются.
- Текущая картина:
  - `unit` покрыт плотно;
  - `smoke` есть;
  - `integration` уже не пуст: есть минимальный `train -> load artifact -> score` roundtrip;
  - проверки публичных CLI и artifact contracts уже есть, но пока в основном живут в `unit`, а не как отдельный формальный `contract`-слой.
- На текущем этапе нет признаков мертвых `test_*.py`, которые не собираются `pytest`.
- Но есть ранний перекос:
  - много unit-тестов на synthetic data;
  - мало сквозных проверок на уровне реальных связок модулей;
  - формальный `contract`-слой пока выражен слабее, чем `unit` и `integration`.

## Debug Plan Для Тестового Слоя

- Перед следующей функциональной волной делаем короткий тестовый cleanup.
- Порядок такой:
  - убрать явное дублирование synthetic builders и test helpers через `conftest.py` или узкий `tests/helpers`;
  - развести тесты по фактическим уровням ответственности, а не только по историческим именам файлов;
  - добавить минимальный `contract`-слой для публичных CLI и artifact contracts;
  - добавить хотя бы один настоящий `integration`-прогон на `train -> load artifact -> score`;
  - не плодить новые тесты, которые проверяют ту же самую ветку логики вторым именем;
  - не добавлять тест ради строки покрытия, если он не ловит отдельный риск.

- При необходимости усилить `priority`-логику инженерными критериями телескопа и наблюдательными ограничениями.
- Перед отдельной `subclass`-волной сделать локальный `data audit` по таблицам БД и только потом решать, нужен ли добор из `Gaia` или внешнего spectral catalog.
- Вернуться к `data engineering` и пересмотреть training source под новый label-contract:
  - `spectral_class`;
  - `spectral_subclass`;

- Когда дойдем до внешней API-границы, отдельно повторно обсудить `FastAPI` и возможный `async` только на краю системы.

## Правило Для Тестового Cleanup

- Не увеличиваем количество тестов механически.
- Каждый новый тест должен отвечать на вопрос:
  - какой риск он ловит;
  - чем он отличается от уже существующих.
- Если несколько тестов используют один и тот же synthetic dataset builder, выносим его в общее место.
- Если проверка проходит только через monkeypatch и не ловит реальный стык модулей, это не integration-тест.
- Если модуль публичный, но проверяется только косвенно, отмечаем это как технический долг до следующего cleanup.

## Правило Для Cleanup CLI

- Не строим дополнительный framework поверх `argparse`.
- Не вводим `service`, `facade`, `manager` и другие искусственные слои без реальной причины.
- `main.py` должен остаться только точкой входа и верхнеуровневого dispatch.
- Логику parser registration и command handlers можно выносить в отдельные модули по ответственности, если это уменьшает размер и связность `main.py`.
- Если правка не упрощает чтение кода и не уменьшает ответственность файла, она не нужна.

## Критерии качества

- один файл = одна ответственность;
- минимально достаточные зависимости;
- типизация по умолчанию;
- `ruff`, `mypy`, `pytest` обязательны;
- после каждой микро-задачи проходит QA gate.

## Рабочий Цикл Разработки

- Пишем код и проверяем его не двумя разными фазами, а одной связкой.
- Базовый цикл такой:
  - блок кода;
  - микропроверка этого блока;
  - сверка с официальной документацией по написанию и проверке;
  - только потом следующий блок.
- После завершения законченного модуля:
  - проводим полную локальную проверку всего связанного контура;
  - дополнительно сверяем спорные места с официальной документацией и принятыми практиками;
  - только потом считаем модуль закрытым.
- После завершения крупной части:
  - проводим повторную всестороннюю проверку уже собранного слоя;
  - если нужно, возвращаемся и упрощаем код до продолжения следующей части.

## Правило Для Написания И Проверки

- Пишем код с оглядкой на официальные рекомендации библиотеки, инструмента или фреймворка, который используем.
- Проверяем код тоже с оглядкой на официальные рекомендации по валидации, конфигурации и expected behavior.
- Если пишем модуль поверх внешнего инструмента:
  - сначала проверяем официальный способ это делать;
  - потом реализуем;
  - потом проверяем по официальному expected behavior.
- Это правило действует не только при исправлении ошибок, но и при обычной разработке.

## Правило диагностики

- не лечим симптомы, если причина еще не найдена;
- перед исправлением сначала воспроизводим проблему и определяем корневую причину;
- если проблема связана с инструментом, библиотекой, конфигом, API или внешним контрактом, сначала сверяемся с официальной документацией или первоисточником;
- если официального источника недостаточно, только тогда используем дополнительные открытые источники;
- в микро-задаче решение считается корректным только если понятно:
  - что именно было причиной;
  - почему выбранное исправление устраняет причину, а не маскирует эффект.

## QA Gate

- проблема воспроизведена;
- корневая причина названа явно;
- при необходимости проверен официальный источник;
- исправление минимально и соразмерно причине;
- после исправления повторно пройдены локальные проверки.
