# Аудит документации и ТЗ по унификации

Дата: 11 марта 2026 года

## 1. Цель

Привести документацию проекта к единому русскоязычному формату:

- у каждого Python-файла должна быть актуальная шапка-модульный docstring;
- у каждой значимой функции, класса, dataclass, TypedDict и CLI-точки входа должен быть docstring в едином стиле;
- в docstring функций должно быть ясно:
  - что делает функция;
  - что она принимает и возвращает;
  - откуда берёт данные или на какие модули/таблицы/артефакты опирается;
  - какие есть побочные эффекты и исключения, если они есть.

Отдельное требование: не писать в модульных шапках исторические или устаревшие описания. Документация должна описывать текущую ответственность файла после рефакторинга.

## 3. Краткий вывод аудита

### 3.1 Что уже хорошо

- Покрытие модульными docstring уже есть практически везде.
- У функций и классов docstring в кодовой базе в основном присутствуют.
- В ряде файлов уже есть хорошие русскоязычные модульные шапки:
  - `src/input_layer.py`;
  - `src/logbooks/program_run.py`;
  - `src/logbooks/decision_layer.py`;
  - часть production- и EDA-модулей.

### 3.2 Основные проблемы

1. Документация неоднородна по языку.
   В проекте есть значительный пласт англоязычных и смешанных docstring. По результатам машинной проверки выявлено 30 файлов и 167 docstring-элементов, где используется не русский язык или смешанный стиль.

2. Есть устаревшие модульные шапки.
   Самый явный пример: `src/gaussian_router.py`. Файл уже является backward-compatible facade, но сверху остаётся большой исторический комментарий, который описывает старую роль монолитного файла.

3. Есть слишком короткие шапки у фасадов.
   Файлы:
   - `src/model_gaussian.py`;
   - `src/star_orchestrator.py`;
   - `src/eda.py`;
   - `src/router_eda.py`;
   - `src/decision_layer_calibrator.py`.

   Сейчас они описаны одной строкой, но не объясняют:
   - что файл делает сейчас;
   - что логика уже вынесена в пакет;
   - какие группы символов он реэкспортирует;
   - зачем файл оставлен в проекте.

4. Функциональные docstring часто описывают только действие, но не источник данных.
   Особенно это заметно в:
   - DB-loader функциях;
   - persist/save функциях;
   - CLI-обвязках;
   - scoring-функциях, где важно объяснить, какие столбцы ожидаются и какие поля возвращаются.

5. Пакетные `__init__.py` местами описаны слишком абстрактно или по-английски.
   Это относится к нескольким публичным пакетам и фасадным точкам входа.

6. Тесты документированы неравномерно.
   Для тестов не нужен тяжёлый format `Параметры/Возвращает`, но язык и стиль должны быть едиными. Сейчас часть helper-функций в тестах описана по-английски.

## 4. Целевой стандарт документации

## 4.1 Общий принцип

Берём за основу:

- `PEP 257` для формы docstring;
- NumPy/научный стиль секций;
- русский язык для описаний.

Исключение:

- устоявшиеся технические термины и имена полей можно оставлять в исходном виде: `host_posterior`, `router_log_posterior`, `run_id`, `source_id`, `parallax_over_error`.

## 4.2 Шаблон для файла

Каждый модуль должен начинаться с docstring в тройных кавычках. Не использовать длинные баннеры из `# =====`, если это можно выразить обычным docstring.

Рекомендуемый формат:

```python
"""Краткое назначение модуля.

Что делает модуль:
    - ...
    - ...

Откуда модуль получает данные:
    - ...
    - ...

Что возвращает или публикует наружу:
    - ...

Что модуль не делает:
    - ...
"""
```

Для фасадов обязательна явная формулировка:

- это backward-compatible facade;
- основная логика находится в конкретном пакете;
- файл нужен для совместимости со старым импортом;
- собственной доменной логики почти не содержит.

## 4.3 Шаблон для функции

Для обычной публичной функции целевой формат такой:

```python
def load_input_candidates(...) -> pd.DataFrame:
    """Загрузить входных кандидатов для production pipeline.

    Параметры
    ----------
    engine : Engine
        Подключение к Postgres.
    source_name : str
        Имя relation, из которой читаются данные.
    limit : int | None
        Ограничение на число строк.

    Возвращает
    ----------
    pd.DataFrame
        DataFrame с одной строкой на `source_id` и минимально
        необходимыми колонками для router и decision layer.

    Источник данных
    ----------
    Читает relation из Postgres и использует introspection helpers
    из `priority_pipeline.relations`.

    Исключения
    ----------
    RuntimeError
        Если relation не существует или не хватает обязательных колонок.
    """
```

Если функция чистая и не тянет внешние данные, достаточно:

- краткого описания;
- `Параметры`;
- `Возвращает`.

Если функция что-то сохраняет, пишет в БД или на диск, обязателен блок:

- `Побочные эффекты`.

Если функция является thin-wrapper над другой функцией, в docstring это нужно явно писать.

## 4.4 Шаблон для класса, dataclass, TypedDict

Для классов и структур данных нужно описывать их роль, а не повторять имена полей. Поля уже типизированы, но должно быть понятно, зачем сущность существует.

Пример:

```python
@dataclass(frozen=True)
class BaseScoringResult:
    """Результат базового preview-прогона до применения калибровки.

    Хранит входной батч, результат router, разделение на ветки и
    host-scoring до применения calibrated decision layer.
    """
```

## 4.5 Правила для тестов

Для тестов действуют отдельные правила:

- модульный docstring обязателен;
- helper-функции получают короткие docstring на русском;
- `test_*` функции могут оставаться короткими, одной фразой;
- не нужно добавлять тяжёлые секции `Параметры/Возвращает` к каждому тесту.

## 4.6 Что запрещено

- исторические простыни, не соответствующие текущей архитектуре;
- смешение русского и английского без причины;
- комментарий-баннер вместо нормального module docstring;
- docstring вида `Run main.` или `Helper function.` без смысла;
- дублирование кода или типов вместо объяснения ответственности.

## 8. Критерии готовности

Работа считается завершённой, когда выполняются все условия:

- во всех модульных шапках используется актуальное описание текущей роли файла;
- в проекте не осталось длинных устаревших баннеров вместо нормального module docstring;
- все публичные docstring в `src/` и `analysis/` русскоязычные;
- DB/file/CLI/persist функции явно описывают источники данных и побочные эффекты;
- фасады явно помечены как фасады и указывают канонический пакет с логикой;
- тестовые docstring приведены к единому русскому стилю без лишней бюрократии;
- после правок проект по-прежнему проходит:
  - `./venv/bin/pytest -q`;
  - `./venv/bin/ruff check src tests analysis`;
  - `./venv/bin/mypy src tests analysis`.
