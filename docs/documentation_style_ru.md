# Стандарт документации проекта

Дата фиксации стандарта: 11 марта 2026 года

## Цель

Этот документ фиксирует единый формат документации проекта, чтобы:

- новые Python-модули и функции сразу оформлялись в одном стиле;
- markdown-документы, журналы и отчёты были сопоставимы между собой;
- не возвращаться к повторной унификации формата в следующих итерациях.

## 1. Общие принципы

- Основной язык документации: русский.
- Имена полей, функций, артефактов и DB relations сохраняются в исходном виде:
  `host_posterior`, `router_log_posterior`, `run_id`, `source_id`.
- Документация описывает текущую ответственность файла или функции, а не историю рефакторинга.
- Фасады совместимости должны прямо называться фасадами совместимости.
- Если функция читает БД, файл или artifact, это нужно писать явно.
- Если функция пишет в БД или на диск, это нужно явно помечать как побочный эффект.

## 2. Стандарт для Python-документации

## 2.1 Модуль

Каждый Python-файл начинается с module docstring.

Рекомендуемый шаблон:

```python
"""Краткое назначение модуля.

Что делает модуль:
    - ...
    - ...

Где находится основная логика или откуда приходят данные:
    - ...

Что модуль не делает:
    - ...
"""
```

Для фасадов обязательно указывать:

- что это фасад или слой обратной совместимости;
- какой пакет является каноническим;
- что новой доменной логики в фасад добавлять не нужно.

## 2.2 Функция

Минимум, который должен содержать docstring функции:

- что делает функция;
- что возвращает;
- если есть внешний источник данных, откуда он берётся;
- если есть запись на диск или в БД, блок `Побочные эффекты`.

Для публичных функций предпочтителен формат со смысловыми блоками:

- `Параметры`
- `Возвращает`
- `Источник данных`
- `Побочные эффекты`
- `Исключения`

## 2.3 Классы и структуры данных

Для `dataclass`, `TypedDict` и прочих структур описывается роль сущности в системе, а не только буквальный набор полей.

## 3. Стандарт для markdown-документов

## 3.1 README и обзорные документы

Обязательный каркас:

1. Что это за проект.
2. Актуальная архитектура.
3. Где находится каноническая логика.
4. Как запускать основные сценарии.
5. Где лежат журналы, артефакты и дополнительные документы.

## 3.2 Журнал прогона программы

Формат файла `run_XXX.md`:

1. Шапка:
   - идентификатор прогона;
   - дата;
   - статус.
2. Контекст запуска.
3. Артефакты и режимы.
4. Параметры decision layer.
5. Размеры потоков.
6. Ключевые метрики.
7. Top-N кандидаты.
8. Предупреждения.
9. Ошибки.
10. Итог.
11. Следующий шаг.

## 3.3 Журнал калибровки decision layer

Формат файла `iteration_XXX.md`:

1. Шапка:
   - идентификатор итерации;
   - дата;
   - статус.
2. Контекст итерации.
3. Что меняем.
4. Формула.
5. Режимы скоринга.
6. Гипотеза.
7. Параметры итерации.
8. Ожидаемый эффект.
9. Фактический результат.
10. Сводка по top-N.
11. Итог.
12. Следующий шаг.

## 3.4 QA-отчёты и отчёты по проверкам

Рекомендуемый каркас:

1. Цель проверки.
2. Входные условия.
3. Результаты прогона.
4. Контрактные проверки.
5. Физические sanity-check.
6. Проверка качества кода.
7. Ограничения.
8. Итоговый вывод.

## 4. Терминология

Используем как базовые:

- `router` — физическая классификация звезды;
- `host-model` — contrastive `host-vs-field` модель;
- `production pipeline` — полный боевой pipeline ранжирования;
- `decision layer` — слой мягких коэффициентов приоритизации;
- `artifact` — сериализуемая модель или файл с параметрами;
- `logbook` — журнал прогонов или калибровки.

## 4.1 Исторические документы

- Исторические заполненные журналы можно править по языку, заголовкам и
  служебным полям.
- Нельзя переписывать фактически использованную формулу, параметры или
  результаты прошлых прогонов только ради унификации терминологии.

## 5. Что считается нарушением стандарта

- английский docstring там, где нет явной причины оставлять английский;
- большой исторический баннер вместо обычного module docstring;
- устаревшие ссылки на удалённые скрипты;
- markdown-шаблон, не соответствующий текущему scoring-контракту;
- описание фасада как основной реализации, если логика давно вынесена в пакет.
