# Соответствие проекта требованиям ВКР

Дата: 13 марта 2026 года

## 1. Назначение документа

Этот документ фиксирует, как текущий проект соотносится с формальными
требованиями ВКР и какие блоки ещё нужно добрать до полностью
закрытой структуры работы.

Документ нужен для двух целей:

- не потерять обязательные разделы пояснительной записки;
- держать единый рабочий список недостающих артефактов в репозитории.

## 2. Основание

Основание для сверки:

- файл `Требования к ВКР.docx`;
- текущее состояние репозитория;
- зафиксированный preprocessing-скрипт из DBeaver;
- действующая архитектура `router -> host-model -> decision layer`.

Важно:

- исходный файл с требованиями является шаблоном под другую предметную область;
- предметная формулировка в шаблоне не совпадает с темой звёзд и экзопланет;
- при этом структура ВКР и перечень обязательных блоков остаются применимыми.

## 3. Краткий вывод

Текущий проект уже хорошо закрывает:

- постановку практической задачи;
- EDA;
- production pipeline;
- README и репозиторную часть;
- воспроизводимую кодовую базу с тестами;
- документирование архитектуры и QA.

Главные незакрытые или неполностью закрытые требования:

- формальное описание приложения как пользовательского контура;
- финальные защитные артефакты: пояснительная записка, PDF и презентация;
- финальная синхронизация repo-артефактов с запиской и слайдами.

## 5. Минимально достаточная конфигурация для ВКР

С учётом текущего состояния проекта и формальных требований, минимально
достаточный набор выглядит так:

1. Основная модель:
   текущая Gaussian `V1` как физически интерпретируемая основная схема.
2. `Baseline 1`:
   упрощённый legacy Gaussian-контур для честного сравнения с текущей `V1`.
3. `Baseline 2`:
   классический ML-baseline, например `RandomForest`.
4. Нейросетевой блок:
   компактный `MLP`, не как production-ядро, а как отдельная
   исследовательская модель для закрытия требования по ИНС.
5. Единый comparative report:
   train/test метрики, ranking-метрики и итоговая таблица сравнения.
6. Formalized preprocessing:
   SQL, ADQL, notebook и markdown-документ по data lineage.

## 6. Что не нужно делать в текущей волне

Для закрытия требований ВКР сейчас не требуется:

- срочно внедрять `GMM/router mixture`;
- срочно внедрять иерархический `router`;
- перестраивать production pipeline ради самой формальной полноты;
- делать GUI, если CLI уже можно описать как прикладной контур.

Эти направления остаются допустимыми как будущая научная эволюция проекта,
но не являются ближайшим обязательным шагом.
