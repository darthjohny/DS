## Зачем нужен этот план

Основной контур проекта уже собран:

- обучающие и прикладные пайплайны работают;
- итоговые артефакты воспроизводимы;
- документация, notebook и регрессионный слой приведены в порядок;
- главный прикладной результат для ВКР уже зафиксирован.

Теперь нужен не новый слой логики, а тонкая витрина поверх существующей
системы. Она должна позволить:

- показать работу проекта без запуска notebook;
- показать качество моделей и ключевые метрики по слоям;
- загрузить внешний `CSV` по контракту проекта и получить результат;
- просмотреть итоговые артефакты, приоритеты и карточки объектов;
- использовать интерфейс в демонстрации и в защите ВКР.

## Цель

Собрать небольшой локальный интерфейс на `Streamlit`, который не дублирует
прикладную логику проекта, а использует уже существующие модули `src/exohost`
и показывает их результат в удобном виде.

Главный принцип:

Интерфейс не принимает архитектурных решений сам по себе. Он либо вызывает
существующий контур `decide`, либо читает уже готовые артефакты.

## Почему выбран Streamlit

`Streamlit` подходит проекту по нескольким причинам:

- работает в том же стеке `Python`, без отдельного фронтенд-контура;
- хорошо показывает таблицы, метрики, графики и карточки объектов;
- позволяет быстро собрать многосекционный интерфейс;
- удобен для локальной демонстрации и для защиты.

В качестве архитектурной основы нужно использовать именно тот multipage-подход,
который сейчас рекомендует официальная документация `Streamlit`: `st.Page` и
`st.navigation`, а не старую схему только через каталог `pages`.

## Что должен уметь интерфейс

### 1. Главный экран

Нужно показать:

- краткое описание проекта;
- схему системы;
- текущий главный прикладной результат;
- ключевые метрики последнего рабочего запуска.

### 2. Запуск на внешнем CSV

Нужно дать пользователю:

- загрузить `CSV` по входному контракту проекта;
- запустить `decide` без ручного CLI;
- получить путь к новому набору артефактов;
- увидеть короткую итоговую сводку по запуску.

### 3. Страница качества моделей

Нужно показать:

- ключевые метрики по `ID/OOD`, `coarse`, `host` и `refinement`;
- размер выборок и состав артефактов;
- калибровочные метрики там, где они уже есть в проекте;
- понятную пометку, какие слои являются сильными, а какие требуют осторожной
  интерпретации.

Эта страница нужна не только для демонстрации, но и как короткий слой доверия
к итоговому списку кандидатов.

### 4. Просмотр готового запуска

Нужно уметь:

- открыть существующий `run_dir`;
- прочитать `metadata.json`, `final_decision.csv`, `priority_ranking.csv` и
  другие рабочие артефакты;
- показать распределения `id / unknown / ood` и `high / medium / low`;
- вывести верхний список кандидатов.

### 5. Карточка объекта

Нужно показать по выбранной строке:

- `source_id`;
- итоговый класс и подкласс;
- причины итогового решения;
- `host_similarity_score`;
- `priority_score`;
- ключевые физические параметры;
- состояние качества и маршрут по пайплайну.

## Что не входит в интерфейс

В этот пакет не входят:

- обучение моделей;
- редактирование policy через UI;
- ручная настройка порогов в браузере;
- работа с локальной БД напрямую;
- перенос исследовательских notebook внутрь интерфейса.

Если позже понадобится более широкий демонстрационный режим, это должен быть
отдельный этап, а не первая версия интерфейса.

## Предлагаемая структура

Нужен отдельный тонкий слой `ui`, который не смешивается с `posthoc`,
`reporting` и `training`.

Предлагаемый контур:

- `streamlit_app.py` — единая точка входа и маршрутизация страниц;
- `src/exohost/ui/contracts.py` — контракты интерфейса;
- `src/exohost/ui/session_state.py` — состояние интерфейса;
- `src/exohost/ui/loaders.py` — загрузка `CSV` и артефактов;
- `src/exohost/ui/run_service.py` — вызов существующего `decide`;
- `src/exohost/ui/components/` — отдельные блоки интерфейса;
- `src/exohost/ui/pages/` — страницы приложения;
- `tests/unit/ui/` — unit-тесты helper-слоя интерфейса.

## Предлагаемый порядок сборки

### Этап 1. Только просмотр готового запуска

Сначала делаем read-only версию:

- домашняя страница;
- страница метрик моделей;
- выбор готового `run_dir`;
- сводки по запуску;
- верхний список кандидатов;
- карточка объекта.

Это даст быструю и устойчивую демонстрацию без риска сломать боевой контур.

### Этап 2. Запуск `decide` из интерфейса

После read-only версии добавляем:

- загрузку `CSV`;
- валидацию входного контракта;
- кнопочный запуск `decide`;
- показ итоговой сводки по новому запуску.

### Этап 3. Полировка

Только потом можно добавлять:

- фильтры;
- экспорт выборок;
- удобные таблицы и небольшие графики;
- косметическую полировку для демонстрации.

## Ожидаемый результат

Пакет считается закрытым, когда:

- приложение локально запускается через `Streamlit`;
- интерфейс читает готовые артефакты без дублирования логики;
- страница метрик моделей показывает качество рабочего контура без notebook;
- пользователь может загрузить внешний `CSV` и получить новый запуск;
- верхний список кандидатов и карточка объекта показываются без notebook;
- код интерфейса остается тонким, модульным и покрытым тестами.

## Как запускать интерфейс локально

Рабочая точка входа интерфейса:

- [streamlit_app.py](/Users/evgeniikuznetsov/Desktop/dspro-vkr/streamlit_app.py)

Минимальный локальный сценарий:

```bash
python -m pip install -r requirements-streamlit-v2.txt
python -m streamlit run streamlit_app.py --server.address 127.0.0.1 --server.port 8501
```

После запуска интерфейс должен быть доступен в браузере по одному из адресов:

- `http://127.0.0.1:8501`
- `http://localhost:8501`

Что важно зафиксировать:

- запуск выполняется из корня репозитория;
- отдельный `PYTHONPATH=src` не нужен, потому что entrypoint сам добавляет
  `src` в `sys.path`;
- если стандартный порт занят, его можно сменить, например на `8502`;
- интерфейс рассчитан на локальную демонстрацию и читает готовые `run_dir`
  из `artifacts/decisions`.
