## Зачем нужен этот документ

Регрессионные тесты страхуют не отдельные функции, а поведение системы на
малых фиксированных наборах данных. Поэтому их нужно запускать осознанно и
отдельно от обычных локальных тестов.

## Базовый запуск

Полный слой регрессионных тестов запускается так:

```bash
.venv-v2/bin/pytest -q tests/regression
```

Это основной способ быстро проверить, что после изменений система ведет себя
так же, как должна.

## Когда нужен полный регрессионный прогон

Полный запуск нужен:

- после изменений в `quality_gate`;
- после изменения порогов итогового ранга наблюдений;
- после изменений в `decide` и итоговых артефактах;
- после правок в обзорных сводках, на которые опираются notebook и выводы.

## Когда достаточно адресного запуска

Если изменение затрагивает только один слой, можно запускать соответствующий
подкаталог:

- `tests/regression/posthoc`
- `tests/regression/decision`
- `tests/regression/reporting`

## Что не заменяет регрессионный слой

Регрессионные тесты не заменяют:

- `unit` для локальной логики;
- `integration` для коротких связок между слоями;
- `smoke` для стартового контура;
- полный рабочий прогон на базе данных.

## Итог

Регрессионные тесты нужны как быстрый и воспроизводимый слой доверия между
локальными модульными тестами и полным рабочим прогоном проекта.
