## Зачем Нужен Отдельный Регресс-Слой

В проекте уже есть:

- плотный `unit`-слой;
- минимальный `smoke`;
- один `integration`-тест.

Этого достаточно для проверки локальной логики и базового запуска пакета.

Но этого недостаточно для ответа на вопрос:

«После изменения кода система в целом все еще ведет себя так, как мы ожидаем?»

Именно на этот вопрос отвечает регресс-слой.

## Роли Тестовых Слоев

### `tests/unit`

Проверяет:

- отдельные функции;
- компактные модули;
- локальные контракты;
- небольшие куски логики.

### `tests/integration`

Проверяет:

- отдельный сквозной путь между несколькими слоями;
- обычно один ограниченный контур, а не весь проект.

### `tests/smoke`

Проверяет:

- что пакет импортируется;
- что базовый CLI-контур поднимается;
- что проект не сломан на самом верхнем уровне.

### `tests/regression`

Проверяет:

- ключевые системные инварианты;
- поведение pipeline на небольших frozen fixtures;
- стабильность `quality_gate`, `priority`, `decide` и artifact bundle;
- отсутствие повторного появления уже закрытых дефектов.

## Что Входит В Регресс-Слой

В `tests/regression` входят только проверки поведения, которые:

- слишком широки для обычного `unit`;
- слишком важны, чтобы их держать только в документации или notebook;
- должны страховать проект от тихого смыслового дрейфа.

## Что Не Входит В Регресс-Слой

В регресс-слой не входят:

- полные боевые прогоны на живой БД;
- performance-бенчмарки;
- архивные исследовательские проверки;
- большие нечитабельные snapshot-дампы;
- дублирование существующих `unit`-тестов.

## Правила Для Fixtures

Frozen fixtures регресс-слоя должны быть:

- маленькими;
- читаемыми;
- локальными для проекта;
- документированными;
- стабильными по смыслу.

Нельзя:

- тащить в fixtures большие реальные выгрузки;
- хранить бессмысленные giant-CSV ради видимости покрытия;
- завязывать регресс на живые внешние сервисы и текущую БД.

## Правила Для Самих Тестов

Регресс-тест должен:

- проверять инвариант поведения;
- падать при реальной смысловой поломке;
- оставаться читаемым;
- иметь шапку файла на русском;
- не зависеть от случайного порядка данных без нужды.

Регресс-тест не должен:

- сравнивать весь большой артефакт до последнего байта;
- требовать постоянного ручного обновления без пользы;
- смешивать несколько больших тем в одном файле.

## Структура Слоя

Регресс-слой строим так:

```text
tests/regression/
├── __init__.py
├── conftest.py
├── fixtures/
├── decision/
├── posthoc/
└── reporting/
```

Смысл подпапок:

- `fixtures` — только замороженные входы и маленькие expected-структуры;
- `decision` — сквозное поведение и artifacts;
- `posthoc` — `quality_gate`, `priority` и policy-level regressions;
- `reporting` — устойчивость summary-слоя, который используют notebook и docs.

## QA Для Регресс-Слоя

Для каждого шага внутри регресс-пакета делаем:

- `ruff`
- targeted `mypy`
- targeted `pyright`
- targeted `pytest`

После появления первого рабочего набора регресс-тестов:

- добавляем отдельный scoped запуск всего `tests/regression`;
- фиксируем это в общей QA-процедуре проекта.
