# Политика Зависимостей Проекта

Этот документ фиксирует, как в проекте разделяются пакетные и локальные
зависимости после cleanup-волны `MTZ-C16`.

## Зачем Нужна Политика

До cleanup-пакета один файл `requirements-v2.txt` одновременно использовался:

- как локальный манифест окружения;
- как источник `project.dependencies` в `pyproject.toml`.

Для стадии быстрого роста это было удобно, но для стабильного проекта это
слишком хрупко:

- пакет получал notebook- и QA-инструменты как runtime-зависимости;
- в один список смешивались библиотечный runtime, визуализация и проверки;
- метаданные пакета зависели от файла, который потенциально может содержать
  pip-специфику.

## На Что Опираемся

- [Setuptools: dependency management](https://setuptools.pypa.io/en/stable/userguide/dependency_management.html)
- [pip: requirements file format](https://pip.pypa.io/en/stable/reference/requirements-file-format/)
- [nbclient docs](https://nbclient.readthedocs.io/en/latest/)

## Принятое Решение

В проекте используются два уровня манифестов:

1. `requirements-runtime-v2.txt`
2. `requirements-v2.txt`

### `requirements-runtime-v2.txt`

Это минимальный runtime-слой installable пакета.

Его свойства:

- используется в `pyproject.toml` как источник `project.dependencies`;
- содержит только прямые runtime-зависимости активного кода;
- не содержит notebook-инструментов;
- не содержит QA-инструментов;
- не содержит заделов “на будущее”.

### `requirements-v2.txt`

Это полный локальный манифест для повседневной работы.

Его свойства:

- включает runtime-слой через `-r requirements-runtime-v2.txt`;
- добавляет notebook- и визуализационные зависимости;
- добавляет QA-инструменты;
- не используется как источник пакетных metadata.

## Что Считается Runtime-Зависимостью

Библиотека остается в runtime-слое, если выполняется хотя бы одно из условий:

- она импортируется активным кодом из `src/exohost`;
- она нужна для исполнения installable CLI-контура;
- она используется как прямая зависимость, а не только как транзитивная.

Примеры:

- `joblib` остается в runtime, потому что код напрямую использует
  `joblib.dump` и `joblib.load`;
- `astropy` остается в runtime, потому что ingestion-слой импортирует
  `astropy.table`.

## Что Считается Локальной Рабочей Зависимостью

Библиотека остается только в `requirements-v2.txt`, если она нужна для:

- notebook-работы;
- визуализации;
- адресного notebook-QA;
- статических проверок и тестов.

Примеры:

- `matplotlib`
- `seaborn`
- `jupyterlab`
- `nbclient`
- `pytest`
- `mypy`
- `ruff`
- `pyright`

## Что Не Держим В Активном Манифесте Без Кода

Если библиотека:

- не импортируется активным кодом;
- не участвует в активном notebook/QA-контуре;
- и является только потенциальным будущим планом,

то она не остается в текущем манифесте “на всякий случай”.

По этой причине из активного dependency-слоя убраны:

- `fastapi`
- `uvicorn`
- `astroquery`
- `nbconvert`

Если позже для них появится рабочий код или воспроизводимый notebook-поток,
они возвращаются отдельным осмысленным изменением.

## Практическое Правило Для Новых Изменений

При добавлении новой библиотеки нужно ответить на два вопроса:

1. Она нужна installable пакету или только локальной разработке?
2. Она импортируется напрямую или живет только транзитивно?

После этого:

- runtime-зависимость идет в `requirements-runtime-v2.txt`;
- локальная рабочая зависимость идет в `requirements-v2.txt`;
- документация по новым правилам обновляется вместе с кодом.
