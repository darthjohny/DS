# Политика Шапок Для Исходных Файлов

Дата фиксации: `2026-04-05`

## Зачем Нужны Шапки

Структурная шапка в начале файла нужна не как декоративный комментарий, а как
короткая навигация по проекту:

- что это за файл;
- какая у него ответственность;
- что в нем не должно лежать;
- какой слой идет следующим.

Это помогает:

- быстрее восстанавливать контекст;
- собирать схему проекта для презентации;
- позже сделать README-оглавление в виде карты слоев.

## Обязательный Шаблон

Для активных `src`- и `tests`-файлов используем такой формат:

```python
# Короткое описание файла.
#
# Этот файл отвечает только за:
# - первая ответственность;
# - вторая ответственность.
#
# Следующий слой:
# - соседний модуль или notebook;
# - слой, который использует этот файл дальше.
```

## Правила

- шапка должна быть на русском;
- первая строка должна кратко и понятно описывать файл;
- блок `Этот файл отвечает только за:` обязателен;
- блок `Следующий слой:` обязателен;
- шапка не должна обещать то, чего в файле нет;
- если файл тестовый, шапка должна явно говорить, что именно он проверяет;
- если файл служебный `__init__.py`, в шапке должно быть сказано, что это
  пакетный marker/export-layer.

## Чего Не Делаем

- не пишем длинные простыни текста;
- не дублируем docstring и шапку одновременно, если достаточно шапки;
- не пишем абстрактные фразы вроде `этот файл важен`;
- не перечисляем всю архитектуру проекта в каждом файле.

## Приоритет Внедрения

Шапки приводим к единому виду пакетами:

1. активный `reporting`-слой;
2. связанные `tests/unit/reporting`;
3. остальные active-пакеты `src/exohost`;
4. остальные active-пакеты `tests/unit`;
5. архивный слой — только по мере необходимости.
