# Контракт качества данных и `OOD` для версии `V2`

## Цель

Этот документ фиксирует ранний слой проверки качества данных и `OOD` для версии `V2`.

Задача документа:

- не пускать плохие объекты в обычный контур молча;
- явно отделить `качество данных` от `астрофизической классификации`;
- заранее определить, где объект считается:
  - пригодным;
  - сомнительным;
  - выходящим за область доверия модели.

## Основной принцип

`quality_state` и `ood_state` проверяются до основной тонкой классификации и до
расчета приоритета.

То есть логика такая:

- сначала качество и доверие к объекту;
- потом метки;
- потом сходство с host-профилем и наблюдаемость;
- потом итоговый приоритет.

## Обязательные сигналы качества

В первой рабочей версии явно учитываем:

- `ruwe`
- `parallax_over_error`
- наличие критически важных признаков
- базовую физическую правдоподобность значений

К критическим признакам относятся минимум:

- `teff`
- `logg`
- `radius`

Если они отсутствуют или явно повреждены, объект не должен идти в обычную классификацию как будто с ним все нормально.

## RUWE

`RUWE` считаем ранним индикатором качества астрометрического решения.

Правило:

- пороговые значения не выдумываем;
- финальное правило для `RUWE` фиксируем по литературе и практике `Gaia`;
- конкретный threshold документируем явно, а не держим в голове.

На уровне архитектуры `RUWE` считается:

- quality feature;
- ранним gate signal;
- частью `observability` только вторично.

## `Parallax over error`

`parallax_over_error` считаем вторым ранним quality-сигналом.

Назначение:

- оценить надежность дистанционной информации;
- не подмешивать сомнительные решения, связанные с расстоянием, в обычное
  ранжирование без флага качества.

## Отсутствие критически важных признаков

Если отсутствуют критические признаки:

- объект не должен автоматически считаться нормальным кандидатом;
- вводим отдельную ветку:
  - `unknown`
  - или `OOD`
  - или жесткий low-priority fallback

Точный путь определяется в реализации, но отсутствие данных нельзя маскировать
под валидный обычный случай.

## Проверка физической правдоподобности

До обычной классификации также проверяем:

- что числовые значения лежат в физически правдоподобных диапазонах;
- что в признаках нет грубых противоречий;
- что объект не выглядит как явный артефакт источника.

Это не заменяет полную OOD-логику, но защищает контур от грубого мусора.

## Разделение `quality_state` и `ood_state`

`quality_state` и `ood_state` не одно и то же.

`quality_state` отвечает на вопрос:

- хороши ли исходные данные и наблюдательные параметры.

`ood_state` отвечает на вопрос:

- находится ли объект в области, где модели вообще имеют право уверенно говорить.

Примеры:

- плохой `RUWE` может ухудшать `quality_state`, но не всегда означает чистый `OOD`;
- экзотический объект с необычным сочетанием признаков может быть `OOD`, даже если quality-поля хорошие.

## Допустимые исходы раннего фильтра

На первой волне архитектурно допускаем:

- `pass`
- `unknown`
- `ood`
- `reject`

В коде конкретные значения могут называться иначе, но смысл должен остаться таким же.

## Чего не делаем

- не считаем `RUWE` декоративным дополнительным признаком;
- не проверяем качество слишком поздно, когда объект уже прошел половину контура;
- не используем `unknown` и `OOD` как синонимы;
- не объясняем плохое ранжирование тем, что «модель так решила», если объект
  изначально был низкого качества.

## Связь с ранжированием

Фильтр `quality/OOD` идет раньше расчета приоритета.

Итоговый слой приоритета должен видеть:

- объект прошел quality gate;
- объект сомнительный;
- объект `OOD`;
- объект отклонен.

Ранжирование не должно превращаться в место, где молча пытаются исправить
плохое качество данных.

## Связь с базой данных и новыми источниками

Для новой data engineering-волны должны быть явно доступны поля:

- `ruwe`
- `parallax_over_error`
- критические физические признаки

Если source этого не дает:

- это должно быть видно в аудите;
- такой источник не считается полноценной таблицей для обучения и опоры без
  оговорок.

## Логика фильтрации для первой рабочей версии

Для первой DB-реализации gate используем только поля, которые уже
зафиксированы в модели данных Gaia DR3 или в наших отдельных источниках `OOD`.

### Сигналы Gaia, на которые мы опираемся

- `ruwe`
  - показатель качества из `gaia_source`
- `parallax_over_error`
  - отношение параллакса к его ошибке из `gaia_source`
- `non_single_star`
  - флаг наличия дополнительной информации в non-single star tables
- `classprob_dsc_combmod_star`
  - вероятность класса single star по DSC-Combmod
- `teff_gspphot`, `logg_gspphot`, `mh_gspphot`
  - параметры GSP-Phot, которые в Gaia DR3 описаны как оценки при предположении,
    что источник является single star
- `radius_flame`, `lum_flame`, `evolstage_flame`
  - официальные поля FLAME из `astrophysical_parameters`

### Минимальный набор критически важных признаков

Для `pass` в обычный поток refinement в первой рабочей версии требуем минимум:

- `teff_gspphot`
- `logg_gspphot`
- `mh_gspphot`
- `bp_rp`
- `parallax`
- `parallax_over_error`
- `ruwe`

`radius_flame` считаем отдельным обязательным сигналом для refinement:

- его отсутствие не делает строку автоматическим `reject`;
- но переводит строку как минимум в `unknown`, а не в чистый `pass`.

### Практические пороги первой рабочей версии

В первой рабочей версии фиксируем следующие пороги проекта:

- `ruwe > 1.4` -> `unknown`
- `parallax_over_error < 5` -> `unknown`
- missing critical core features -> `reject`
- missing `radius_flame` -> `unknown`
- `non_single_star > 0` -> `candidate_ood`
- `classprob_dsc_combmod_star < 0.5` -> `candidate_ood`
- membership в отдельном OOD-pool -> `ood`

Важная оговорка:

- значения `1.4`, `5` и `0.5` здесь являются правилами проекта для первой
  рабочей версии;
- это не жесткие пороги из официальной документации Gaia;
- официальная документация задает смысл полей, а не один обязательный общий cut.

### Выходы первой рабочей версии

На первой DB-волне используем:

- `quality_state`:
  - `pass`
  - `unknown`
  - `reject`
- `ood_state`:
  - `in_domain`
  - `candidate_ood`
  - `ood`

### Таблица для `unknown` и разбора спорных случаев

Отдельная таблица `lab.gaia_mk_unknown_review` должна содержать все строки, где:

- `quality_state <> 'pass'`
  или
- `ood_state <> 'in_domain'`

Это нужно, чтобы:

- не терять сомнительные строки;
- не пускать сомнительные `OOD`-объекты в обычное обучение и расчет;
- иметь отдельный аналитический контур разбора.

## Критерий готовности контракта

Контракт считается зафиксированным, если:

- любой новый источник можно проверить на готовность по качеству и `OOD`;
- загрузчики и контур обработки понимают, что фильтр идет раньше классификации;
- ранжирование не маскирует проблемы качества данных.

## Решение после первой калибровки

После первой калибровки на рабочей таблице `lab.gaia_mk_quality_gated`
правила первой рабочей версии оставляем без изменений.

Фактическая картина:

- baseline:
  - `pass`: `178439` (`44.36%`)
  - `unknown`: `63823` (`15.87%`)
  - `reject`: `159964` (`39.77%`)
- relaxed:
  - `pass`: `195815` (`48.68%`)
  - `unknown`: `46447` (`11.55%`)
  - `reject`: `159964`
- strict:
  - `pass`: `162741` (`40.46%`)
  - `unknown`: `79521` (`19.77%`)
  - `reject`: `159964`

Вывод:

- главная причина для `reject` — отсутствие основных признаков;
- смягчение threshold-политики дает ограниченный выигрыш по `pass`;
- `unknown` на первой волне оставляем как отдельный слой разбора;
- базовые пороги сохраняем до отдельной волны интеграции host-модели и
  приоритета.

Это решение соответствует архитектурному принципу:

- не проталкивать сомнительные строки в обычный `id`-контур силой;
- не путать низкокачественные объекты с обычными внутридоменными объектами;
- держать `unknown/review` как отдельный аналитический контур.
