# Семантика целевой host-задачи

## Цель

Этот документ фиксирует, что именно должен предсказывать host-слой в новой
волне проекта и чего он предсказывать не должен.

Задача документа:

- убрать двусмысленность вокруг формулировки "вероятность наличия планеты";
- явно разделить:
  - семантику исходных данных из `NASA Exoplanet Archive`;
  - нашу проектную целевую задачу;
- не позволить слою приоритета опираться на нестрогую интерпретацию
  host-оценки.

## Что важно в семантике NASA

### `ps`

`ps` хранит planetary-system solutions так, что каждая строка представляет
один planet+stellar+system solution для одной ссылки.

Это удобно как опора для положительной метки confirmed planets, потому что:

- таблица хранит подтвержденные решения по планетным системам;
- строка согласована на уровне одной ссылочной записи;
- для одной звезды и одной планеты может быть несколько ссылочных строк.

### `pscomppars`

`pscomppars` делает одну более полную строку на планету, но документация
прямо предупреждает, что она может быть:

- составлена из нескольких ссылок;
- не полностью согласована для одной системы.

Вывод для проекта:

- `pscomppars` годится для обогащения и демографического обзора;
- `pscomppars` не должен быть единственным источником истины для положительной
  метки.

### `stellarhosts`

Документация для `stellarhosts` прямо указывает, что таблица включает:

- звезды с подтвержденными планетами;
- звездные параметры, не привязанные к конкретному планетному решению;
- гравитационно связанные звезды без планет в составе планетных систем.

Вывод для проекта:

- `stellarhosts` нельзя механически считать чистой таблицей положительных
  host-меток;
- она подходит как источник дополнительных звездных параметров и
  идентификаторов;
- положительная метка должна быть привязана к подтвержденной host-опоре, а не
  ко всем строкам `stellarhosts`.

## Решение проекта: что предсказывает host-слой

На новой волне host-слой предсказывает не
"вероятность существования планеты вообще".

Он предсказывает:

- `host-likeness`
  или
- `confirmed-host prior`

относительно нашей clean confirmed-host population и matched field population.

То есть смысл host-оценки такой:

- насколько объект похож на звезды из нашей обучающей выборки confirmed-host;
- а не какова истинная астрофизическая вероятность того, что вокруг звезды
  есть планета вообще.

## Чего host-слой не должен обещать

Host-слой не должен выдавать это как:

- глобальную вероятность существования любой планеты;
- частоту встречаемости планет в популяции;
- вероятность обнаружения планеты конкретной программой наблюдений;
- вероятность наличия именно каменистой планеты;
- финальный приоритет последующих наблюдений без учета класса, качества и
  наблюдаемости.

Все эти интерпретации слишком сильные для текущей задачи и источников.

## Clean Positive Label Semantics

Положительная метка первой чистой host-волны должна означать:

- звезда относится к популяции confirmed-host, опирающейся на подтвержденные
  записи о планетах из `NASA Exoplanet Archive`.

Практический принцип:

- опорные положительные объекты берем из таблиц confirmed-planet
  (`ps`-семантика);
- `stellarhosts` используем как дополнительный звездный источник;
- `pscomppars` используем как источник для описания и обогащения, а не как
  основной источник меток.

## Clean Negative Label Semantics

Отрицательная метка первой чистой host-волны не означает:

- доказанное отсутствие планет.

Он означает:

- согласованную полевую популяцию, которая в текущем источнике host-данных не
  отмечена как confirmed-host.

То есть `field` здесь:

- рабочий отрицательный класс для задачи классификации;
- а не физическое утверждение "у этой звезды точно нет планет".

## Связь с текущим кодом

Текущая first-wave задача в коде называется:

- `host_field_classification`

И по смыслу это уже близко к правильной постановке:

- positive = `host`
- negative = `field`

Но во всех документах, ноутбуках и последующей интерпретации надо
использовать более аккуратную интерпретацию:

- `host_similarity_score`
- `confirmed-host prior`

а не "вероятность наличия планеты".

## Как это должно выглядеть в выводе

Корректные формулировки:

- "объект имеет высокий host-like score"
- "объект близок к confirmed-host population"
- "confirmed-host prior высокий/средний/низкий"

Некорректные формулировки:

- "вероятность того, что у звезды есть планета, равна X"
- "модель доказала, что звезда не имеет планет"

## Связь со слоем приоритета

Слой приоритета не должен напрямую трактовать host-оценку как конечную
вероятность.

Правильная схема:

- `class priority`
- `host_similarity_score`
- `observability`

объединяются в объяснимый итоговый приоритет.

То есть host-слой является одним из сигналов в интеграции приоритета,
а не самодостаточным итоговым ответом.

## Решение для первой рабочей версии

Для следующего рабочего пакета фиксируем:

- target semantics = `confirmed-host prior / host-likeness`;
- positive label source не берем "как есть" из `stellarhosts`;
- `pscomppars` не используем как primary positive truth;
- ноутбуки и итоговая интерпретация больше не должны называть host-оценку
  "вероятностью наличия планеты".

## Критерий готовности

Документ считается зафиксированным, если:

- известно, что именно предсказывает host-слой;
- известно, какие формулировки запрещены;
- дальше можно проектировать чистый контракт признаков без смыслового дрейфа.
