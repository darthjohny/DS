# Контракт меток MK для `V2`

## Цель

Этот документ фиксирует целевой контракт меток для волны взросления `V2`.

Задача документа:

- не хранить звезду как одну хрупкую метку вида `G2V`;
- разделить MK-классификацию на составные части;
- заранее зафиксировать, что именно войдет в первую волну моделей, а что осознанно откладывается.

## Основной принцип

Внутри ядра проекта не используем строку вида `G2V` как основной target.

Рабочий контракт строим по составным частям:

- `spectral_class`
- `spectral_subclass`
- `luminosity_class`
- `peculiarity_suffix`

Строка полного MK-label допускается только как:

- внешний формат импорта;
- внешний формат экспорта;
- человекочитаемое представление в отчетах.

## Составные части MK-схемы

### 1. Spectral Class

Назначение:

- задает температурный класс верхнего уровня.

Поддерживаемые значения:

- `O`
- `B`
- `A`
- `F`
- `G`
- `K`
- `M`

### 2. Spectral Subclass

Назначение:

- уточняет положение внутри температурного класса.

Поддерживаемые значения:

- `0..9`

Пример:

- `G0` горячее, чем `G5`
- `G5` горячее, чем `G9`

Важно:

- `spectral_subclass` не хранится как `G5`;
- `G` и `5` храним раздельно:
  - `spectral_class = G`
  - `spectral_subclass = 5`

### 3. Luminosity Class

Назначение:

- задает размер и эволюционное положение звезды в схеме MK.

Поддерживаемый полный набор:

- `0`
- `Ia+`
- `Ia`
- `Iab`
- `Ib`
- `I`
- `II`
- `III`
- `IV`
- `V`
- `VI`
- `VII`

На первой волне допускается укрупнение в `luminosity_group`, если:

- данных по полной схеме недостаточно;
- это улучшает устойчивость первой модели;
- укрупнение зафиксировано в docs и явно воспроизводимо.

### 4. Peculiarity Suffix

Назначение:

- хранит химические и спектральные особенности.

На первой волне:

- не используем как основной target;
- сохраняем как опциональный import field;
- не включаем в базовый benchmark.

## Первая волна целевых меток

Для первой новой волны `V2` считаем обязательными:

- `spectral_class`
- `spectral_subclass`
- `luminosity_class` или осознанно зафиксированную `luminosity_group`

Опциональными и отложенными:

- `peculiarity_suffix`

## Что Не Делаем

- не склеиваем `spectral_class`, `spectral_subclass` и `luminosity_class` в один target;
- не используем `G2V` как внутреннюю основную метку;
- не смешиваем labels с quality-признаками;
- не подменяем `luminosity_class` старым `dwarf/evolved` без явной схемы преобразования.

## Связь со старым `V2`

Старая схема:

- `spectral_class`
- `evolution_stage`

Новая схема:

- `spectral_class`
- `spectral_subclass`
- `luminosity_class`
- при необходимости отдельно `luminosity_group` или `stage`

Это не отменяет старый слой сразу, а расширяет его.

### Отображение для старого `evolution_stage`

Для совместимости со старым `V2` coarse `evolution_stage` допускается только как
производное поле поверх `luminosity_class`, а не как замена MK-контракта.

Явная схема преобразования:

- `V`, `VI`, `VII` -> `dwarf`
- `0`, `Ia+`, `Ia`, `Iab`, `Ib`, `I`, `II`, `III`, `IV` -> `evolved`
- `NULL` или неподдержанный luminosity code -> `NULL`

Правило:

- `evolution_stage` хранится только как совместимое поле для старых
  загрузчиков и представлений;
- основной MK-слой все равно остается на `luminosity_class`;
- mapping должен жить в коде как отдельная маленькая функция, а не размазываться по SQL.

## Как это влияет на модели

Новые задачи должны быть раздельными:

- `spectral_class classification`
- `spectral_subclass classification`
- `luminosity_class` или `luminosity_group classification`

Не допускается:

- учить одну модель на случайной строковой смеси `G2V`;
- путать `luminosity_class` с итоговым приоритетом для телескопа.

## Как это влияет на БД

В новых таблицах и view храним отдельно:

- `spectral_class`
- `spectral_subclass`
- `luminosity_class`
- `peculiarity_suffix`, если источник его дает

Старые имена таблиц не трогаем.

Новые имена таблиц используем только для новой волны.

## Критерий готовности контракта

Контракт считается зафиксированным, если:

- любой новый источник можно проверить на соответствие полям;
- любой загрузчик понимает, какие метки обязательны;
- benchmark и training не зависят от монолитной строки `G2V`.
