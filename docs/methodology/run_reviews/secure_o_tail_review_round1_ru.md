# Secure O Tail Review Round 1

## Цель

После вынесения спорного `OB`-пула в отдельный review-контур проверить, что
осталось в надежной группе `O`.

Задача этого шага:

- понять, похожа ли эта группа на реальные `O`-обозначения, а не на остаточный
  parser-noise;
- проверить, что говорит по ней hot-star слой Gaia;
- зафиксировать, можно ли уже сейчас делать по ней агрессивные правила.

## Что Проверяли

Источник:

- `lab.gaia_ob_secure_o_like_subset`

Для traceability подтягивали исходные спектральные обозначения из:

- `lab.gaia_mk_external_labeled`

Опора по hot-star semantics:

- `spectraltype_esphs`
- `flags_esphs`
- `teff_esphs`

из official Gaia DR3 `astrophysical_parameters`.

## Live Результат

Надежный локальный хвост `O` сейчас очень маленький:

- `n_rows = 11`

Из них:

- `n_with_numeric_subclass = 2`
- `n_with_teff_esphs = 1`
- `n_in_gold_sample_oba = 2`

Медианы:

- `median_teff_gspphot ≈ 15522.56 K`
- `median_lum_flame ≈ 1096.27`

Важно:

- `median_teff_esphs` на текущем шаге неинформативна как сводка, потому что
  численная `teff_esphs` есть только у одной строки.

## Исходные Метки

В этой группе больше нет широких ambiguous `OB...` меток.

Остались именно `O`-подобные формы:

- `O`
- `O4V`
- `O(He)`
- `O(H)3/4`
- `O(H):III/V(e)`
- `O(H)9I`
- `O8I/III`
- `O7.5I/III`
- `O(H)4/5V`
- `O(H)8If`

То есть parser-induced широкий шум по `OB...` уже не просачивается в этот слой.

## Что Говорит Gaia

Для всех `11` строк:

- `spectraltype_esphs = 'O'`
- `esphs_class_letter = 'O'`

Но при этом численные hot-star параметры неполны:

- только `1` строка имеет непустую `teff_esphs`
- у большинства строк hot-star модуль подтверждает класс `O`, но не дает
  численной температуры

Это значит:

- признак горячего класса у Gaia здесь есть;
- но жесткое температурное правило по `teff_esphs` на этом хвосте сейчас строить
  нельзя.

## Сравнение С Чистыми Наборами O И B

Для sanity-check этот хвост был сравнен с уже существующими чистыми reference
наборами:

- `public.gaia_ref_class_o`
- `public.gaia_ref_class_b`
- `public.gaia_ref_evolved_class_o`
- `public.gaia_ref_evolved_class_b`

Важно:

- это не новый внешний источник;
- это тот эталонный набор, который уже использовался в coarse-ветке;
- он нужен здесь именно как опорная точка для сравнения.

По `teff_gspphot` картина такая:

- `secure_o_tail`: `median ≈ 15522.56 K`
- `reference_b`: `median ≈ 12825.10 K`
- `reference_evolved_b`: `median ≈ 12355.91 K`
- `reference_o`: `median ≈ 30145.72 K`
- `reference_evolved_o`: `median ≈ 34604.62 K`

Абсолютная разница по температуре:

- до `reference_b`: `≈ 2697.46 K`
- до `reference_evolved_b`: `≈ 3166.64 K`
- до `reference_o`: `≈ 14623.16 K`
- до `reference_evolved_o`: `≈ 19082.06 K`

То есть по `teff_gspphot` этот хвост заметно ближе к эталонному `B`, чем к
эталонному `O`.

Это не отменяет исходные `O`-метки и hot-star tag от Gaia, но подтверждает, что
текущий маленький хвост остается физически пограничным случаем.

## Вывод

Главная parser-проблема уже снята:

- спорный `OB`-пул вынесен отдельно;
- оставшийся локальный `O`-хвост больше не похож на смесь ambiguous `OB...`.

Но этот хвост:

- очень маленький;
- неполный по численным hot-star параметрам;
- по `teff_gspphot` заметно ближе к чистому `B`, чем к чистому `O`;
- поэтому пока не годится для грубых ручных правил по температуре.

## Следующий Шаг

Следующий шаг уже не про broad `OB`-pool, а про эту узкую группу:

1. отдельно проверить, как текущая coarse-модель ведет себя именно на этих `11`
   строках после parser-fix;
2. если этого окажется мало, уже потом решать:
   - нужен ли узкий внешний спектроскопический источник;
   - или хватает текущего контура и review-policy.
